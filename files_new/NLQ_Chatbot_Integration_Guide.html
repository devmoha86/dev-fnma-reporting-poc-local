Section 1

What we are building

A context-aware NLQ chatbot panel that slides in when the user clicks the chat icon on any chart or KPI card. It already knows what that chart shows, what data feeds it, and what questions are answerable — before the user types a word.

💬

Icon-triggered chat panel

Each chart card and KPI card has an NLQ icon button. Clicking it opens a slide-in chat panel scoped to that specific chart's data and API endpoint.

📋

Session + dashboard context

The panel receives the logged-in user's session (analyst name, servicer filter state) and the chart's metadata — title, chart type, fields, API source — as context on every message.

📡

Live data awareness

The NLQ backend knows what each chart is rendering via chart_nlq_context.json and can call the same FastAPI endpoints to answer questions with current data.

✨

Smart suggestions

Suggested prompts seeded from context — "show as table", "add loan balance column", "filter to SVC-001" — make first interactions instant and discoverable.

Section 3

User flow — icon click to answer

Eight steps from the user clicking the chat icon to seeing a contextualized answer in the panel.

1

User hovers a chart card — NLQ icon appears

The chat icon button (💬) is hidden by default and appears on :hover of the chart card. Each button carries a data-chart-id attribute matching the chart_id in chart_nlq_context.json.

2

Click fires openNlqChat(chartId) in Angular component

The Angular dashboard component calls NlqService.getChartContext(chartId). The service has already loaded chart_nlq_context.json from src/assets/nlq/ at app startup — this is a synchronous local lookup, no network call needed at click time.

3

Angular reads current session + filter state

The component captures: logged-in user (Chi Nicholas), active dashboard filters (Servicer Number = All, Servicer Name = All), and the current chart render state (which servicers are visible, current date range). These are bundled into the context payload.

4

NLQ chat panel slides in with pre-seeded suggestions

The panel opens showing the chart title, a scoped header ("Asking about: Delinquency Rate Trend"), and 3–4 suggestion chips drawn from chart_nlq_context.json → suggested_followups[]. User can tap a chip or type freely.

5

User submits a question — Angular posts to NLQ backend

The Angular service POSTs to the NLQ backend (POST /api/nlq/query) with: the user's text, the chart_id, the system_prompt_context from the JSON, and the active filter state. The NLQ backend uses system_prompt_context as a system message prefix so its LLM stays scoped to the chart's data.

6

NLQ backend resolves intent → constructs FastAPI call

Using metadata.json (nlq_alias → field mapping) and openapi.json (param names), the NLQ backend builds the correct GET URL: e.g. GET /api/rejected-events/by-servicer?servicer_id=SVC-001&activity_period=2026-Q1.

7

FastAPI returns JSON — NLQ formats the answer

Your rejected_events.py runs apply_filters() + Pandas groupby("servicer_id") and returns a JSON array. The NLQ backend formats this as a natural-language answer, optionally with a rendered table or chart suggestion.

8

Answer renders in the NLQ panel with follow-up suggestions

The chat panel shows the answer alongside smart follow-up options: "Show as table", "Add loan balance column" (if available in API), "Filter to this servicer only", "Export CSV".

Section 5

The two context files the NLQ panel reads

These two JSON files are the contract between your Angular app and the NLQ backend. Both are generated by Copilot prompts and committed to the repo.

M

metadata.json

Answers: "what can a user ask?"
Contains data sources, column nlq_aliases, filterable/aggregatable flags, dashboard_charts[] registry, kpi_cards[], computed_fields[].

📁 docs/nlq/metadata.json

C

chart_nlq_context.json

Answers: "what is this chart showing right now?"
Contains per-chart system_prompt_context, answerable questions, unanswerable questions, and suggested_followups loaded on icon click.

VS Code Copilot prompts to execute

Prompt A — Generate the Angular NlqService

Creates the Angular service that loads chart context, manages session state, and communicates with the NLQ backend.

PROMPT A NlqService — Angular 18
@workspace Generate a complete Angular 18 service at [angular-app/src/app/services/nlq.service.ts]. The service must: 1. Load [angular-app/src/assets/nlq/chart_nlq_context.json] once on init via HttpClient, cache it with shareReplay(1), and expose getChartContext(chartId: string) returning the matching ChartNlqContext object. 2. Capture current dashboard session state via a setSessionContext() method that accepts: - userName: string (e.g. "Chi Nicholas") - activeFilters: { servicer_number: string, servicer_name: string } 3. Expose sendNlqQuery(payload: NlqQueryPayload): Observable that POSTs to: http:///api/nlq/query The NlqQueryPayload type must include: - chart_id: string - user_query: string - session: { userName, activeFilters } - chart_context: { system_prompt_context, api_endpoint, filters_supported, answerable_questions } - active_filters: { servicer_number, servicer_name } 4. Expose loadChartSuggestions(chartId: string): Observable returning the suggested_followups[] array from the cached context for that chart_id. Use environment variable NLQ_BASE_URL from [angular-app/src/environments/environment.ts]. Add proper error handling with catchError returning an empty NlqResponse on failure. Export all interfaces (ChartNlqContext, NlqQueryPayload, NlqResponse).

Prompt E — Add /api/nlq/context endpoint to FastAPI

Adds a metadata endpoint to your backend so the NLQ developer can fetch chart context dynamically instead of reading static files.

PROMPT E FastAPI — NLQ context endpoint
@workspace Add a new file [backend/nlq_context.py] that registers two new FastAPI routes: Route 1: GET /api/nlq/context Returns the full contents of [docs/nlq/chart_nlq_context.json]. Tags: ["NLQ"]. Add a docstring: "Returns chart-level NLQ context for the Angular chatbot panel." Route 2: GET /api/nlq/context/{chart_id} Returns the single chart_context entry matching chart_id from the JSON. Raises HTTPException 404 if chart_id not found. Tags: ["NLQ"]. Route 3: GET /api/nlq/metadata Returns the full [docs/nlq/metadata.json]. Tags: ["NLQ"]. Docstring: "Returns source-to-target metadata vocabulary." Import pattern — follow the same pattern used in [backend/rejected_events.py]: - Use APIRouter not app directly - Load JSON files at module level using pathlib.Path - Return dicts (FastAPI auto-serialises to JSON) Register the router in [backend/main.py] with: from nlq_context import router as nlq_context_router app.include_router(nlq_context_router) After adding: the NLQ developer can bootstrap via: GET http://:8000/api/nlq/context GET http://:8000/api/nlq/metadata GET http://:8000/api/nlq/context/delinquency_rate_trend

Section 9

Smart suggestion engine — rules for each chart

The NLQ backend should apply these rules when generating suggestions after each response. These are seeded from the confirmed dashboard data and API capabilities.

Chart Suggestion trigger Suggestion text shown to user What it does
All charts After any answer with row data "Show as table" Calls same API endpoint, renders response as HTML table in chat
All charts If available_columns_not_shown[] is non-empty "Add [column name] column" Re-calls endpoint, adds the field to table or chart tooltip
Delinquency Trend After showing all-servicer data "Filter to SVC-001 Apex only" Calls ?servicer_id=SVC-001
Delinquency Trend After servicer-level answer "Compare all servicers side by side" Calls without servicer_id filter, returns full groupby
Loan Count by Region After regional data shown "Which region has the most loans?" Sorts by loan_count DESC, highlights top region
Portfolio Balance After balance data shown "Show as % of total portfolio" Computes each servicer balance / $1,525M total
KPI cards After KPI value shown "Break this down by servicer" Calls /api/rejected-events/by-servicer to disaggregate
All charts Always in suggestions "Export this data as CSV" Returns blob download of current API response as CSV
Unanswerable guard: Before calling the NLQ backend, the Angular component checks the user's input against unanswerable_questions[] in chart_nlq_context.json using simple substring matching. If matched, it renders a local "This data is not available because..." message without a network call. This prevents the NLQ backend from hallucinating answers to questions the API cannot support.
