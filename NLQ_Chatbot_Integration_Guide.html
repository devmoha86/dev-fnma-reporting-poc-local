Section 2

What we are building

When the user clicks the NLQ icon in the top-right of the dashboard, the Strands NLQ window opens pre-loaded with the full dashboard context — every chart, every KPI, all active filters, and all API endpoints — so the user can ask any question about what they are currently seeing.

🎯

One icon, full dashboard context

The NLQ icon at top-right opens a window aware of all charts and KPIs simultaneously — Delinquency Trend, Loan Count, Portfolio Balance, Status Distribution, and all three KPI cards.

🗂️

Filter state is the scope

The active filter bar values (Servicer Number, Servicer Name, Analyst Name) at click time narrow what the NLQ answers. Selecting SVC-003 before clicking means every Strands tool call uses SVC-003 as the default filter.

🧠

Strands + Bedrock powers answers

The NLQ backend uses the Strands framework with the org AWS Bedrock API key. Your FastAPI endpoints are registered as Strands tools. The NLQ app has its own UI — Angular just launches the window.

🚧

Hard context boundary

Questions outside the dashboard's data scope return a fixed message: "This is outside the current dashboard context — a new query would need to be built." No hallucination.

Section 4

User flow — filter → click NLQ icon → window opens

Eight steps. The NLQ icon is clicked once at the dashboard level — not per chart.

1

User selects filters from the dashboard filter bar

User picks Analyst Name, Servicer Number(s), or Servicer Name from the left sidebar and filter bar, then clicks Submit / Apply. Angular stores: activeFilters = { analyst_name, servicer_numbers[], servicer_name }. The charts re-render with filtered data.

2

User clicks the NLQ icon — top-right corner of the dashboard

The existing icon cluster in the top-right of the content area includes the NLQ button. Clicking it fires openNlqWindow() in the Angular component. No hover needed — this is a persistent visible button, always accessible from any state of the dashboard.

3

Angular captures the full dashboard state at click time

The handler reads: the current dashboard_id (from the dropdown — "Operations Dashboard" or the route), all active filter values, the logged-in user, and the reporting_base URL. It does not need to know which chart the user was looking at — the context covers all of them.

4

Angular encodes context into the NLQ URL and opens the window

window.open(nlqUrl, 'nlq_chat', 'width=860,height=740') launches the Strands NLQ app. The URL carries dashboard_id, active filter state, user name, and your FastAPI base URL. The full context JSON is fetched by the NLQ app on load from your /api/nlq/context/{dashboard_id} endpoint.

5

Strands NLQ window loads — fetches full dashboard context JSON

The Strands app reads the URL params and calls GET {reporting_base}/api/nlq/context/{dashboard_id} to get the complete nlq_context.json for that dashboard. It builds the combined system prompt covering all charts and KPIs, renders the context strip, and shows suggestion chips drawn from across all suggested_followups[].

6

User asks a question — boundary check runs first

Before any Bedrock call, the NLQ app checks whether the question matches any entry in the combined unanswerable_questions[] from all chart contexts. If matched → local boundary message, no API cost. If clear → Strands routes to Bedrock with the full dashboard system prompt.

7

Strands selects the right tool and calls your FastAPI

Bedrock decides which Strands tool answers the question — get_rejected_events_summary() for KPI questions, get_rejected_events_by_servicer() for servicer-level detail. The active filter values from the URL params are passed as default query params. The FastAPI call goes to your machine IP on port 8000.

8

Answer rendered with follow-up suggestions covering the full dashboard

The Strands UI shows the natural-language answer. Follow-up chips cover the breadth of the dashboard: "Show Portfolio Balance breakdown", "Compare all servicers on delinquency", "Show Status Distribution as table", "Add loan count column". The user can pivot across any chart topic without reopening the window.

Section 6

Context handoff — URL payload from Angular to Strands

What Angular encodes in the URL when the NLQ icon is clicked. The Strands app fetches the full JSON on load — the URL only needs the minimal bootstrap params.

URL params (lean — Strands fetches the rest)

// URL opened by window.open() — Strands fetches full context JSON via API // http://NLQ-DEV-IP:NLQ-PORT/nlq?... { // Which dashboard the user clicked NLQ from "dashboard_id": "operations_dashboard", // or "rejected_events" // Filter state from the dashboard at click time "analyst_name": "Chi Nicholas", "servicer_number": "SVC-003", // "All" if no filter selected "servicer_name": "Cardinal Servicing Group", // Base URL of the Reporting Engine so Strands can call it AND fetch context "reporting_base": "http://192.168.1.45:8000", // Strands fetches the full nlq_context.json from this on load: // GET {reporting_base}/api/nlq/context/{dashboard_id} }
Why not pass the full JSON in the URL? The complete nlq_context.json for a dashboard covering 7 components would make the URL too long. Instead, the Strands app uses reporting_base + dashboard_id to fetch the full context from your FastAPI's GET /api/nlq/context/{dashboard_id} endpoint. This also means the context is always fresh — if you update the JSON and restart the backend, the NLQ app gets the latest version on next open.

Angular — openNlqWindow() implementation

Angular — NLQ icon click handlerTS
// In reporting-dashboard.component.ts
openNlqWindow(): void {
  // Read current dashboard ID from route or dropdown value
  const dashboardId = this.currentDashboardId;  // 'operations_dashboard' or 'rejected_events'

  // Capture filter state at click time
  const params = new URLSearchParams({
    dashboard_id:    dashboardId,
    analyst_name:    this.filters.analystName    ?? 'All',
    servicer_number: this.filters.servicerNumber ?? 'All',
    servicer_name:   this.filters.servicerName   ?? 'All',
    reporting_base:  environment.reportingEngineUrl,  // 'http://YOUR-IP:8000'
  });

  const nlqUrl = `${environment.nlqEngineUrl}/nlq?${params.toString()}`;
  const win = window.open(nlqUrl, 'fannie_nlq', 'width=860,height=740,resizable=yes');

  if (!win) {
    // Popup blocked — show snackbar
    this.snackbar.open(
      'Allow popups for this site to open the NLQ assistant.',
      'Dismiss', { duration: 5000 }
    );
  }
}

3. Dynamic tool generation — one tool per unique api_endpoint in context JSON

Instead of hardcoding tools, Strands reads the api_endpoint field from every chart entry in the context JSON and auto-generates one tool per unique endpoint. This means when a new chart is added to the dashboard and the context JSON is updated, the NLQ app automatically gets a new tool — no code change needed.

Strands — dynamic tool generation from context JSON api_endpoint fieldsPY
from strands import tool, Agent
from strands.models.bedrock import BedrockModel
import httpx, re

def build_tools_from_context(bootstrap: dict) -> list:
    """
    Reads chart_contexts[] from the dashboard context JSON.
    Groups charts by their api_endpoint value.
    Generates one Strands @tool function per unique endpoint.
    The tool docstring lists every chart that uses that endpoint,
    so Bedrock knows which charts each tool can answer questions about.
    """
    ctx            = bootstrap["context_json"]
    reporting_base = bootstrap["reporting_base"]
    default_svc    = bootstrap["servicer_number"]

    # Step 1 — group chart_contexts by api_endpoint
    endpoint_to_charts: dict[str, list] = {}
    for chart in ctx["chart_contexts"]:
        ep = chart["api_endpoint"]
        endpoint_to_charts.setdefault(ep, []).append(chart)

    # Step 2 — generate one tool per unique endpoint
    generated_tools = []

    for endpoint, charts in endpoint_to_charts.items():
        chart_names  = [c["chart_title"] for c in charts]
        chart_fields = [f for c in charts for f in c.get("key_fields", [])]
        full_url     = f"{reporting_base}{endpoint}"

        # Build a descriptive docstring Bedrock uses for tool selection
        docstring = (
            f"Fetches data from {endpoint}. "
            f"Powers these dashboard charts: {', '.join(chart_names)}. "
            f"Returns fields including: {', '.join(set(chart_fields))}. "
            f"Use this tool to answer questions about any of those charts, "
            f"including cross-chart comparisons between them."
        )

        # Derive tool function name from endpoint path
        # e.g. /api/rejected-events/by-servicer → tool_rejected_events_by_servicer
        fn_name = "tool_" + re.sub(r"[^a-z0-9]+", "_", endpoint.strip("/").lower()).strip("_")

        # Build the tool function dynamically using a closure to capture endpoint/defaults
        def make_tool(ep_url, doc, svc_default):
            @tool
            def dynamic_tool(
                servicer_id:     str = "",
                activity_period: str = "",
                region:          str = ""
            ):
                params = {}
                # Apply dashboard filter as default when no explicit servicer requested
                sid = servicer_id or (svc_default if svc_default != "All" else "")
                if sid:             params["servicer_id"]     = sid
                if activity_period: params["activity_period"]  = activity_period
                if region:          params["region"]          = region
                return httpx.get(ep_url, params=params, timeout=10).json()

            dynamic_tool.__name__ = fn_name
            dynamic_tool.__doc__  = doc
            return dynamic_tool

        generated_tools.append(make_tool(full_url, docstring, default_svc))

    return generated_tools


# Init agent — tools come entirely from the context JSON, not hardcoded
tools = build_tools_from_context(bootstrap)

agent = Agent(
    model=BedrockModel(
        model_id="anthropic.claude-3-5-sonnet-20241022-v2:0",
        region_name="us-east-1"
    ),
    system_prompt=build_dashboard_system_prompt(bootstrap),
    tools=tools
    # Bedrock sees all tools and their docstrings.
    # For cross-chart questions it calls multiple tools in sequence or parallel.
    # Adding a new chart to nlq_context.json automatically adds a new tool
    # on the next window open — no code change needed.
)
